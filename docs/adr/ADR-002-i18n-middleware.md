# ADR-002: Internationalisation Middleware — next-intl

| Field | Value |
|---|---|
| **Status** | Accepted |
| **Date** | 2024-01-15 |
| **Deciders** | Hein Aung |

---

## Context

SportsMatch Tokyo targets two primary markets:

- **Japan (Tokyo)** — Japanese speakers are the dominant user base.
- **International** — English is the lingua franca for expat users and language-exchange session participants.

Internationalisation (i18n) therefore must be a first-class concern, not an afterthought. Requirements:

1. **URL-based locale routing** — locale should be visible in the URL path (`/en/sessions`, `/ja/sessions`) for SEO and shareability.
2. **No client-side flicker** — locale resolution must happen at the server/edge level, before HTML is sent to the browser.
3. **Automatic redirect** — visiting `/sessions` (no locale prefix) should redirect to the correct locale transparently.
4. **Type-safe translation keys** — typos in translation key names should be caught at compile time.
5. **Compatibility with Next.js App Router** — the solution must work with React Server Components (RSC), where React context is unavailable.

---

## Decision

We will use **next-intl** with its **Next.js Middleware** integration to handle locale detection, redirection, and message loading.

### How it works

#### 1. Middleware (`middleware.ts` → generated by `next-intl`)

The `createNextIntlMiddleware` factory wraps Next.js middleware. On every incoming request it:

1. Reads the `NEXT_LOCALE` cookie (set on first visit or by the language switcher).
2. Falls back to the `Accept-Language` request header.
3. Defaults to `en` if neither is present.
4. Rewrites the URL to prepend the resolved locale segment (`/en/…` or `/ja/…`).
5. Sets the `NEXT_LOCALE` cookie for subsequent requests.

```
GET /sessions
  ↓ middleware
  → detects locale = "ja" (from cookie)
  → rewrites to /ja/sessions
  → sets cookie NEXT_LOCALE=ja
```

#### 2. Route structure (`app/[locale]/…`)

All user-facing pages live under `app/[locale]/`. The `[locale]` dynamic segment receives `en` or `ja` and is validated against the allowed locales list.

#### 3. Message loading (`i18n.ts`)

```typescript
// i18n.ts
export default getRequestConfig(async ({ requestLocale }) => {
  let locale = await requestLocale;
  if (!locale || !locales.includes(locale as Locale)) {
    locale = defaultLocale;
  }
  return {
    locale,
    messages: (await import(`./messages/${locale}.json`)).default,
  };
});
```

Messages are loaded per-request on the server and injected into the component tree via `NextIntlClientProvider`. Only the strings for the active locale are sent to the browser.

#### 4. Usage in components

```typescript
// Server Component or Client Component
import { useTranslations } from 'next-intl';

export function SessionCard() {
  const t = useTranslations('Sessions');
  return <h2>{t('joinButton')}</h2>;
}
```

The `useTranslations` hook is available in both server and client components — next-intl detects the rendering context automatically.

## Alternatives Considered

### 1. `next-i18next`

**Pros:** Battle-tested; large community; compatible with Pages Router.  
**Cons:** Designed for the Pages Router; lacks first-class support for React Server Components; requires `serverSideTranslations` HOC wiring on every page — incompatible with the App Router layout model.

### 2. Native Next.js i18n routing (`next.config.js` `i18n` key)

**Pros:** Zero dependencies; built into Next.js.  
**Cons:** Removed from the App Router. The `i18n` config key in `next.config.js` is only supported by the legacy Pages Router.

### 3. `react-i18next` + manual URL handling

**Pros:** Extremely flexible; framework-agnostic.  
**Cons:** Does not integrate with Next.js Middleware for server-side locale resolution; translations would be loaded client-side causing a flash of untranslated content (FOUC) on first render; no built-in RSC support.

### 4. `Lingui`

**Pros:** Excellent developer experience; message extraction tooling.  
**Cons:** Requires a separate compilation step (message extraction) that adds complexity to the CI pipeline; App Router integration is still experimental.

---

## Consequences

### Positive

- **Zero FOUC** — locale is resolved in middleware before the first HTML byte is sent; there is never a flash of the wrong language.
- **SEO-friendly** — each locale has a distinct URL (`/en/`, `/ja/`); search engines index them independently.
- **RSC-compatible** — `useTranslations` and `getTranslations` work in both Server and Client Components without any provider gymnastics.
- **Type safety** — with the `createTranslator` approach and TypeScript, missing keys throw type errors at build time.
- **Minimal bundle impact** — only the active locale's JSON file is included in the server response; no client-side locale switching library is needed.

### Negative / Trade-offs

- **next-intl version coupling** — the middleware API changed significantly between v3 and v4; upgrades require care.
- **All translatable strings must be extracted** — adding a new UI string requires corresponding entries in both `messages/en.json` and `messages/ja.json` before the build will succeed without warnings.
- **Cookie dependency** — locale persistence relies on the `NEXT_LOCALE` cookie. Users who clear cookies or use incognito mode fall back to `Accept-Language` header detection, which may surprise users who previously selected Japanese.
